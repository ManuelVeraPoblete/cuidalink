## Project Overview

CuidaLink is a home-caregiver mobile app. A primary caregiver manages medications, vital signs, and daily logs for patients, and can invite collaborators who have limited write access.

**Repo structure (being built — only docs exist today):**
```
cuidalink/
├── backend/    ← Java 17 + Spring Boot 3.2, Maven
├── appmovil/   ← React Native 0.74 + TypeScript
└── docs/       ← Design spec and implementation plans
```

## Backend Commands

```bash
cd cuidalink/backend

# Build
mvn compile -q

# Run all tests
mvn test -q

# Run a single test class
mvn test -Dtest=AuthServiceTest -q

# Run the app (requires PostgreSQL running locally)
mvn spring-boot:run

# Package
mvn package -q
```

**Database:** PostgreSQL 15 running locally (`localhost:5432`, database `cuidalink`). No Docker.

**Test database:** Testcontainers spins up PostgreSQL automatically when running integration tests — no manual setup needed.

## Backend Architecture — Hexagonal (strict)

Each of the 6 modules (`auth`, `patient`, `medication`, `vital`, `notification`, `report`) has its own hexagon:

```
com.cuidalink.<module>/
├── domain/
│   ├── model/       ← Java records/classes — NO Spring, NO JPA imports
│   ├── port/in/     ← Use case interfaces (inbound ports)
│   ├── port/out/    ← Repository/service interfaces (outbound ports)
│   └── service/     ← Use case implementations (annotated @Service, OK here)
└── adapter/
    ├── in/rest/     ← @RestController + DTOs
    └── out/
        ├── persistence/   ← JPA entities + repository adapters
        ├── firebase/      ← FCM / FirebaseVerifier adapters
        └── pdf/           ← iText 7 adapter
```

**Hard rule:** `domain/model/` and `domain/service/` packages must never import `jakarta.persistence`, `org.springframework.data`, or any Firebase class. All JPA entities live exclusively in `adapter/out/persistence/`.

**Authorization pattern:** Ownership is checked in the service layer, not the controller. Before any mutation, the service calls `patient.isOwner(requesterId)` or `patient.hasAccess(userId)` and throws `IllegalArgumentException` if denied. The `GlobalExceptionHandler` maps this to HTTP 400.

**API prefix:** `server.servlet.context-path=/api/v1`

## Mobile Architecture — Hexagonal

```
src/
├── domain/
│   ├── entities/      ← Plain TypeScript interfaces (no Axios, no AsyncStorage)
│   ├── repositories/  ← Interface contracts (e.g. PatientRepository.ts)
│   └── usecases/      ← Interfaces grouped by module
├── data/
│   ├── http/          ← Axios instance + Bearer token interceptor
│   └── repositories/  ← ApiXxxRepository.ts — implement domain interfaces
└── presentation/
    ├── navigation/    ← RootNavigator → AuthNavigator / AppNavigator (Bottom Tab)
    ├── stores/        ← authStore.ts (Zustand — auth state ONLY)
    ├── hooks/         ← useInjection.ts (DI: wires repositories to use cases)
    └── screens/       ← Organized by module (auth, home, patients, medications, vitals, profile)
```

**Dependency rule:** `presentation/` imports only from `domain/`. It never imports from `data/` directly. All wiring happens in `useInjection.ts`.

**State split:**
- Zustand: auth state only (current user, Firebase token)
- TanStack Query: all remote data, `staleTime: 60_000`
- Forms: React Hook Form + Zod schemas

## Key Domain Rules

- Only `primaryCaregiver` (owner) can create/edit medications, define vital signs, generate reports, and manage collaborators.
- Collaborators can: confirm/miss medication logs, record vital measurements.
- `InvitationCode`: 8 uppercase alphanumeric chars, expires 24h, single-use.
- `MedicationLog` escalation: if still `PENDING` 30 min past `scheduledAt`, cron changes status to `ESCALATED` and sends FCM push to owner.
- Daily medication logs are generated by cron at 00:01 for each active medication.
- `healthCondition` and `allergies` fields must **never** appear in logs or error messages.
- Report range maximum: 90 days. Only owner can generate.
- `VitalRecord` is immutable once created — to correct, add a new record.

## Testing Strategy

**Backend:**
- Unit tests (JUnit 5 + Mockito): test service classes in isolation, no Spring context.
- Integration tests (Testcontainers): use `@SpringBootTest` + `@ActiveProfiles("test")` — Testcontainers provides real PostgreSQL via `@DynamicPropertySource`.
- Controller tests: `@WebMvcTest` with MockMvc, mock the use case beans.

**Mobile:**
- Unit tests (Jest): domain use cases and data mappers.
- Component tests (RNTL): screens with repositories mocked.
- E2E (Detox): login, confirm medication, record vitals.
